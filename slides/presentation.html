Pillar 01 / 03 · Metrics
Numbers about your system,
over time.
How many? How fast? How often? Metrics tell you what’s normal - so you can spot what’s weird.
How Prometheus works
Push to DataDog.
Pull with Prometheus.
Most monitoring · push
  • Configure endpoints and API keys
  • Your app handles failures and retries
  • Network hiccup → lost data
Prometheus · pull
  • Just expose a /metrics route
  • No credentials, no external config
  • Prometheus handles scraping & retries
Your app doesn’t know Prometheus exists. It just exposes counters and moves on.
In practice
what Prometheus actually sees
// GET /metrics # HELP tile_cache_hits_total # TYPE tile_cache_hits_total counter tile_cache_hits_total 1247 # HELP tile_cache_misses_total # TYPE tile_cache_misses_total counter tile_cache_misses_total 89 tile_cache_age_hours 47.3
# prometheus.yml scrape_configs: - job_name: 'encroach' scrape_interval: 15s static_configs: - targets: - 'localhost:3000'
The core difference
Aggregate at write time, not read time.
Logs

Store every individual event.
Query later.

“What happened at 3:47 AM?”

Prometheus

Store aggregates.
Query summaries.

“What’s the average over the last hour?”

What Prometheus can’t answer
If you didn’t instrument it upfront, the data doesn’t exist.
  • ✕“Show me all requests over 1 second.” - no individual data stored
  • ✕“What’s the average for runner 847?” - already aggregated across all runners
  • ✕“What was the slowest request yesterday?” - individual values not stored
What you can track
Three shapes of number.
Counter
only goes up
tile_cache_hits_total 1,247
requests, errors, hits
Gauge
up and down
tile_cache_age_hours 47.2
cache age, queue depth
Histogram
distributions
http_request_duration_
seconds_bucket{le="0.1"} 892
p50, p95, p99 latency
The tradeoff
Fast forever, if you planned ahead.
Upside
  • Scales to billions of data points
  • Constant memory usage
  • Fast queries, always
Downside
  • Must plan ahead
  • Can’t ask new questions about old data
  • Less flexible than logs
For our cache problem - hits, misses, age. Perfect for Prometheus.

Root cause
Claims land in Postgres.
The cache is never invalidated.
After a claim is written, no one calls s.rdb.Del(ctx, cacheKey).
So every territory read returns the same snapshot from 47 hours ago. The database has the truth. The cache never asks for it.
The fix · one line
not a Go fix - a logic fix
func (s *Service) ClaimTiles(ctx context.Context, runnerID int64, tiles []Tile) error { // ... write claims to Postgres ... if err := s.db.InsertClaims(ctx, runnerID, tiles); err != nil { return err } // Invalidate cache so next GetTerritory rebuilds from DB return s.rdb.Del(ctx, cacheKey).Err() }

The shift
Stop watching for failure.
Start watching for weirdness.
Symptom alerts · reactive
  • “Errors are happening”
  • “Users are complaining”
  • “Workers are failing”
…after the damage.
Behavior alerts · proactive
  • “Cache is never invalidated after claims”
  • “Cache is getting stale”
  • “Hit rate is suspiciously high”
…before runners notice.
prometheus/alerts.yml · three rules
What should be happening - and isn’t.
01 · hit rate
Tile cache not refreshing
tile_cache_hits_total / (hits + misses) > 0.99 for: 1h
A healthy cache has misses. None for an hour = stuck.
02 · cache age
Tile cache stale
tile_cache_age_hours > 6
Our bug pushed this to 47h. This alert fires at 6.
03 · never expires
Cache never rebuilt
increase(tile_cache_misses_total[6h]) == 0
Zero cache misses in 6 hours means invalidation stopped. Page me.

Alert fatigue
Would I mass-notify users about this?

Page-worthy

  • “We stopped talking to our data source”
  • “Data is visibly stale to users”
  • “Core business function is degraded”

Dashboard-only

  • Cache hit rate (interesting, not urgent)
  • Request rate fluctuations
  • Individual slow requests
01
Anti-pattern 01
“No errors” = no problems.
Zero panics. Zero failed goroutines. Zero error logs. Runners getting stale territory for days.
Instead, track success
  • Tiles refreshed in last hour
  • Fresh data served to runner
  • Run processed and territory visible
02
Anti-pattern 02
Caching without an invalidation strategy.
The write path and the cache path were written by two different people. Neither thought the other one would handle it.
Instead, make it explicit
  • Invalidation is part of the feature, not an afterthought
  • Log every cache write and every invalidation
  • Alert on “no invalidations in N hours during active writes”
03
Anti-pattern 03
Tracking hits and misses, not age.
100% hit rate sounds great - until you realise the cache is two days old.
Instead, track age
  • tile_cache_age_hours - a gauge
  • 2 hours average - healthy
  • 47 hours average - definitely not healthy
Anti-pattern 04 · high-cardinality labels
a very bad day for your Prometheus
// Don't do this: tile_cache_hits_total{runner_id="1"} tile_cache_hits_total{runner_id="2"} tile_cache_hits_total{runner_id="3"} // × 10,000 runners × 5 metrics = Prometheus explodes

Aggregate patterns. Rate, p99, by endpoint.
